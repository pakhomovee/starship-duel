One JSON object per line

Write a bot in any language

Your bot is a plain program. The engine writes one JSON request to its stdin per action and reads one JSON reply from its stdout. That's the whole interface — no framework, no network, no dependencies.

📨 Line-delimited JSON ⏱ 2 s per action 🐍 Python or C++ ✅ Legal moves precomputed
01 · The loop

How a match drives your process

Your program is started once and stays alive for the whole game, so it can keep its own memory between decisions.

Lifecycle

  • Spawned once per game. Globals, belief state and opponent models persist across every request.
  • One request per action — not per turn. A turn is at least 2 actions, so you are asked at least twice per turn.
  • Read a line, write a line. Always flush after writing, or the engine will see nothing and time out.
  • stderr is free. It is not parsed, so use it for logging.

A turn, concretely

You start a turn with 2 actions (plus any banked with Overcharge). Each one is a separate request/reply round-trip. The turn ends when you spend them all or reply END_TURN:

→ request  actions_remaining: 2
← reply    {"action":"CLAIM"}

→ request  actions_remaining: 1
← reply    {"action":"JUMP","target":"Veyra"}

… rival's turn — you are not asked …
02 · Limits

Time limit and failure handling

Being slow costs you a move. Crashing costs you the game.

2 seconds per action

The limit is 2 s of wall-clock per action, measured from the moment the request is written to the moment your reply line arrives. It is not a per-turn or per-game budget — every action gets its own fresh 2 s.

Watch your startup. The first request is written immediately after your process launches, so heavy imports or model loading eat into the first action's 2 s. Load lazily, or keep startup well under a second.

What happens when things go wrong

Event Consequence
Reply takes longer than 2 s Strike. The engine plays END_TURN for you (or the first legal action) and the match continues.
Reply is not valid JSON Strike — same substitution.
Reply is an unknown or currently illegal action Strike — same substitution.
Process exits, crashes, or closes stdout Immediate loss. An uncaught exception is a forfeit — catch your own errors and return a legal action instead.
Strikes are logged but never end a live match, however many you collect — you simply keep losing moves. They only matter at upload, where a bot that misbehaves on more than half its moves is rejected.
03 · Input

The request (engine → bot)

One object per line, on stdin. Everything you are entitled to know is here — never more.

{
  "type": "act",               // always "act"
  "turn": 12,                  // turn number (the cap is 200)
  "your_id": 1,                 // 0 or 1 — which ship you are
  "map_id": "reference",
  "domination_target": 80,      // control points needed to win
  "skirmish": 1,
  "campaign_score": [0, 1],       // skirmishes won, indexed by ship id

  "you": {
    "position": "Aurelia Binary",
    "cloaked": false,
    "deep_cloak_turns_left": 0,
    "energy": 1,
    "banked_overcharge": 0,
    "actions_remaining": 1,
    "lives": 3,
    "domination": 24,
    "unlocked": {"proximity_alert": false,
                 "long_range_scanners": true,
                 "jamming": false}
  },

  "rival": {
    "known_position": "Kestrel Binary",  // null unless known for certain
    "last_seen": "Kestrel Binary",       // last confirmed system, or null
    "moves_since_seen": 0,              // hops it could have made since
    "last_turn_actions": ["UNKNOWN", "CLAIM"],  // its whole last turn, in order
    "last_action": "CLAIM",             // the last entry above, or null
    "lives": 3,
    "domination": 31,
    "unlocked": {"proximity_alert": false, …}
  },

  "map": {
    "adjacency": {"Solantis": ["Veyra", "Kestrel Binary"], …},
    "binary_systems": ["Aurelia Binary", "Halcyon Binary"]
  },

  "systems": {
    "Solantis": {
      "owner": null,           // 0, 1, or null
      "owner_known": true,     // false ⇒ "owner" is fogged, not unowned
      "status": "STABLE",       // STABLE | DESTABILIZING | SUPERNOVA
      "binary": false,
      "collapse_in": 30,       // plies to supernova; null = not scheduled
      "cache": {"kind": "ENERGY", "value": 10}   // or null
    }, …
  },

  "legal_actions": [
    {"action": "JUMP", "target": "Kestrel Binary"},
    {"action": "HOLD"}, {"action": "CLAIM"},
    {"action": "SCAN"}, {"action": "END_TURN"}
  ]
}

Your ship — private

FieldMeaning
positionThe system you are in.
cloakedWhether your position is currently hidden from the rival.
deep_cloak_turns_leftTurns of Deep Cloak protection remaining; 0 when not deep-cloaked.
energySpendable energy.
banked_overchargeExtra actions saved for later turns.
actions_remainingActions left in this turn, including this one.
livesLives left. At 0 you are eliminated.
dominationYour control points, raced to domination_target.
unlockedWhich permanent upgrades you own.

The rival — public only

FieldMeaning
known_positionTheir exact system, but only while it is known for certain. Otherwise null.
last_seenThe last system they were confirmed in, or null if never.
moves_since_seenUpper bound on single hops they could have made since. 0 while currently known.
last_turn_actionsEvery action of their last turn, in order. Ones you could not identify read UNKNOWN (they were hidden) or JAMMED (masked by their Jamming) — so the length still tells you how many actions they spent.
last_actionThe final entry of last_turn_actions, for convenience.
livesTheir remaining lives.
dominationTheir control points. Both scores are public.
unlockedWhich upgrades they have bought.

Their energy, banked actions and hidden position are never sent.

04 · Output

The reply (bot → engine)

Exactly one JSON object on stdout, followed by a newline and a flush.

By name

{"action": "JUMP", "target": "Veyra"}
{"action": "CLAIM"}

Action names are case-insensitive and whitespace is trimmed.

By index

{"index": 0}   // = legal_actions[0]

Picks straight out of the legal_actions array you were sent. The simplest possible valid bot.

target is honoured for JUMP only. It is discarded for every other action. In particular a long-range snipe is a plain {"action":"FIRE"} — the engine picks the target itself.

Every action you can send

Send Cost What it does
{"action":"JUMP","target":"X"}freeMove to an adjacent system.
{"action":"HOLD"}freeStay put and slip back under cloak.
{"action":"CLAIM"}freeTake the system you are standing on.
{"action":"FIRE"}3 ⚡Raid your current system.
{"action":"SCAN"}freePin the rival's exact system.
{"action":"DEEP_CLOAK"}14 ⚡Become undetectable for 2 turns.
{"action":"OVERCHARGE"}6 ⚡Bank +1 action for next turn.
{"action":"UNLOCK_PROXIMITY_ALERT"}6 ⚡Permanent defensive unlock.
{"action":"UNLOCK_LONG_RANGE_SCANNERS"}10 ⚡Permanent offensive unlock.
{"action":"UNLOCK_JAMMING"}8 ⚡Permanent masking unlock.
{"action":"END_TURN"}freeStop early and bank spare actions.

Everything currently affordable and legal is already listed in legal_actions, so you never have to re-derive the rules — but the engine validates your reply independently either way.

05 · Hidden information

What the request deliberately withholds

This is a fog-of-war game, and the protocol is the fog. Two things you must infer yourself.

Where is the rival?

You get known_position only when it is certain. The "could be here" candidate set is not provided — rebuild it yourself from the two seeds you are given:

candidates ≈ systems within
  rival.moves_since_seen hops
  of rival.last_seen

A breadth-first search over map.adjacency is enough. Narrow it further using what you can see: a system you are standing next to and know is empty can be struck off.

Who owns what?

The ownership map is fogged too. This is the trap worth spelling out:

The request saysIt means
owner: 0 / 1That ship holds it. Confirmed.
owner: null
owner_known: true		Genuinely unowned. Confirmed.
owner: null
owner_known: false		You have never sensed this system. It may quietly belong to the rival.

A rival claiming under Deep Cloak grows their empire invisibly, so treating fogged systems as free real estate is how you lose the domination race without ever seeing it happen.

06 · Get started

A working bot in ten lines

Both SDKs do nothing but the read/parse/write loop, so you can also skip them and speak the protocol directly.

Python — starship_sdk.py

from starship_sdk import run

def decide(req):
    me = req["you"]
    # fire if we know the rival is right here
    if req["rival"]["known_position"] == me["position"]:
        return {"action": "FIRE"}
    return req["legal_actions"][0]

run(decide)

Upload a single .py file. Copy starship_sdk.py beside it, or inline the six-line loop.

C++ — starship_bot.hpp

#include "starship_bot.hpp"
using nlohmann::json;

json decide(const json& req) {
    if (req["rival"]["known_position"] == req["you"]["position"])
        return json{{"action", "FIRE"}};
    return req["legal_actions"][0];
}
int main() { return starship::run(decide); }

Upload a single .cpp file — it is compiled on the server with -std=c++17 -O2. nlohmann/json is provided.

Submitting

Upload one file on the Tournament page. It is statically scanned, then smoke-tested against random inside the same sandbox it will compete in. If it builds, survives the game, and misbehaves on no more than half its moves, it becomes your active entry and starts playing the ladder.

Test locally first — point the engine at your program directly:

python -m starship_duel.run --bot0 heuristic --bot1 "cmd:./my_bot"
